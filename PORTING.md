# Tidal Connect — Porting to HiFiBerry OS NG

## Upstream

Based on [TonyTromp/tidal-connect-docker](https://github.com/TonyTromp/tidal-connect-docker).
The project packages proprietary Tidal Connect ARM32 binaries inside a Docker container.
Two binaries run inside the container:

- `speaker_controller_application` — volume/transport UI (runs in tmux)
- `tidal_connect_application` — the actual Tidal Connect endpoint (uses PortAudio for audio output)

Both are 32-bit ARM (armhf), dynamically linked against legacy Debian Stretch libraries.

## Why Docker is required

The binaries depend on libraries that do not exist in Debian Trixie (arm64):

| Library | Version | Notes |
|---------|---------|-------|
| libssl1.0.0 | 1.0.1t | Removed after Stretch |
| libcurl3 | 7.38.0 | Conflicts with libcurl4 in Trixie |
| libavformat57 | 3.2.x | Trixie ships libavformat59+ |
| libncursesw5 | 5.x | Trixie ships ncurses6 |
| libportaudio2 | 19.6.0 | Needs armhf build |

Docker provides the legacy Raspbian Stretch environment these binaries expect.

## Changes from upstream

### 1. Removed `apt-get upgrade` from Dockerfile

The upstream Dockerfile runs `apt-get upgrade` inside the `raspbian/stretch` container.
On an arm64 host, this triggers a systemd postinst script that segfaults (exit code 139)
because systemd's ARM32 binary misbehaves under the kernel's AArch32 compatibility layer.

Since we only need specific libraries (not a fully upgraded Stretch system), the upgrade
step is unnecessary and was removed.

### 2. Kernel: switched to 4K page size (`kernel8.img`)

The Raspberry Pi 5 / CM5 default kernel (`kernel_2712.img`) uses **16K pages**. The ARM32
shared libraries in the Docker container have ELF LOAD segments aligned to 4K pages. The
kernel rejects these with:

```
libsystemd.so.0: ELF load command alignment not page-aligned
```

There is no container-level workaround — page size is a kernel property that affects all
processes. The fix is to use the standard aarch64 kernel with 4K pages:

```
# /boot/firmware/config.txt
kernel=kernel8.img
```

The ~5% performance difference from 16K vs 4K pages is negligible for an audio appliance.

**This is a system-wide change.** If Tidal Connect is packaged for general distribution,
the installer should set this automatically on Pi 5/CM5 hardware and document the trade-off.

### 3. Audio routing through PipeWire (PulseAudio socket)

This was the most significant adaptation. The upstream project accesses ALSA hardware
directly (`/dev/snd`). On HiFiBerry OS NG, PipeWire owns the ALSA device exclusively —
PortAudio inside the container finds zero devices when PipeWire is running.

**Approaches considered:**

| Approach | Verdict |
|----------|---------|
| Direct ALSA (`/dev/snd`) | Fails — PipeWire holds device exclusively |
| Stop PipeWire when Tidal plays | Works but breaks all other players |
| ALSA loopback (`snd-aloop`) | Works but fixed format, no clock sync, extra latency |
| PulseAudio TCP | Works but requires opening a TCP port |
| **PulseAudio Unix socket** | **Chosen** — clean, no kernel modules, automatic format negotiation |

**How it works:**

```
Container:  PortAudio -> ALSA API -> alsa-pulse plugin -> libpulse (armhf)
                                                              |
                                              Unix socket (wire protocol)
                                                              |
Host:                                              PipeWire-pulse -> PipeWire -> DAC
```

The PulseAudio wire protocol is architecture-independent — a 32-bit client connects to
a 64-bit server over the same socket with no issues. This is fundamentally different from
shared libraries (like the PipeWire ALSA plugin), which must match the architecture.

**Container-side changes:**

- Installed `libpulse0` and `libasound2-plugins` (armhf, from Stretch repos)
- Created `/etc/asound.conf` routing ALSA default through PulseAudio:
  ```
  pcm.!default { type pulse }
  ctl.!default { type pulse }
  ```
- Set `PULSE_SERVER` to point at the host's PulseAudio socket

**Host-side requirements:**

- `pipewire-pulse.service` and `pipewire-pulse.socket` must be running
- The PulseAudio socket at `/run/user/<UID>/pulse/native` is mounted into the container
- The PulseAudio cookie at `~<user>/.config/pulse/cookie` is mounted read-only for auth

**docker-compose.yml volumes** (generated by install script with actual paths):

```yaml
volumes:
  - /run/user/<UID>/pulse/native:/run/user/<UID>/pulse/native
  - <HOME>/.config/pulse/cookie:/root/.config/pulse/cookie:ro
```

### 4. Trimmed unnecessary packages

Removed from the Dockerfile install: `git`, `neovim`, `zsh`, `portaudio19-dev`. These are
development/editor tools not needed at runtime. Saves ~100MB in the image.

### 5. Metadata bridge (`tidal-bridge.sh`)

New script (not in upstream) that bridges Tidal Connect metadata to ACR (AudioControl).
It scrapes the `speaker_controller_application` tmux output inside the container and:

- POSTs `song_changed`, `state_changed`, `position_changed` events to ACR's player API
  at `http://localhost:1080/api/player/tidal/update`
- Writes `/tmp/tidal-status.json` with current state
- Syncs volume from the Tidal app's volume bar to the ALSA Digital mixer

This uses ACR's generic player event API — no ACR code changes were needed.

## File layout on device

```
/opt/tidal-connect/
  docker-compose.yml      # Container configuration
  .env                    # FRIENDLY_NAME, PLAYBACK_DEVICE, MQA settings
  entrypoint.sh           # Starts speaker_controller + tidal_connect_application
  tidal-bridge.sh         # Metadata bridge to ACR
  tidal-watchdog.sh       # Container health monitor
  Docker/
    Dockerfile            # Container image definition
    src/bin/              # Proprietary ARM32 binaries
    src/id_certificate/   # Certificate (required for Tidal auth)
    src/licenses/         # License files
    entrypoint.sh         # Original entrypoint (copied into image at build)

/etc/systemd/system/
  tidal-connect.service   # Manages Docker container
  tidal-bridge.service    # Metadata bridge (depends on tidal-connect)
  tidal-watchdog.service  # Health monitor (depends on tidal-connect)
```

## Known limitations

- **Certificate**: Uses the upstream project's certificate for Tidal authentication.
  This has licensing implications for distribution.
- **ARM32 only**: The proprietary binaries are 32-bit ARM. No aarch64 or x86 versions exist.
- **Requires 4K page kernel on Pi 5/CM5**: System-wide kernel change needed.
- **PipeWire user detection**: The install script reads `/etc/hifiberry.user` to find the
  audio user and generates docker-compose.yml with the correct UID and home directory.
- **pipewire-pulse must be running**: If it stops (e.g., after a PipeWire restart), the
  container loses audio. The service dependency chain does not currently ensure this.
